Second-order line-search minimization

Newton-CG tutorial

Newton-CG is a practical way to get Newton-like steps on large smooth problems. It teaches the optimizer about curvature without forcing you to store, factorize, or invert the full Hessian matrix.

Why Newton-CG Exists

Suppose you want to minimize a smooth function \(f(x)\). The gradient \(g_k=\nabla f(x_k)\) points uphill, so \(-g_k\) points downhill. That is enough to make progress, but not always enough to make efficient progress. On a long, curved valley, steepest descent can spend many steps crossing the valley walls instead of moving along the valley floor.

Newton's method uses more information. It asks: "if the objective near \(x_k\) were a quadratic bowl, where would the bottom of that bowl be?" The local quadratic model is

\[ \begin{aligned} m_k(p) &= f(x_k) + g_k^\mathsf{T}p\\ &\quad + \frac{1}{2}p^\mathsf{T}H_kp,\\ H_k&=\nabla^2 f(x_k). \end{aligned} \]

Here \(p\) is not a point in the original problem. It is a proposed movement away from \(x_k\). The gradient term \(g_k^\mathsf{T}p\) predicts first-order slope. The Hessian term \(\frac{1}{2}p^\mathsf{T}H_kp\) predicts how that slope bends as you move.

The bottom of this quadratic model satisfies \(\nabla m_k(p)=g_k+H_kp=0\), which gives the Newton equation:

\[ H_kp_k=-g_k. \]

A textbook Newton method might solve that system directly. Newton-CG solves it approximately with conjugate gradients. That swap is the whole idea: keep the useful curvature of Newton's method, but replace a dense matrix solve with an iterative solve that only needs products of the form \(H_kv\).

Mental model: Newton-CG is not "gradient descent plus a fancy line search." It is "fit a local quadratic, approximately find the bottom of that quadratic, then use a line search to decide how far to move in the real objective."

The Algorithm Map

Newton-CG has two nested pieces. The outer loop is a line-search optimizer. The inner loop is a conjugate-gradient solve of the Newton system at the current point.

  1. Start at \(x_k\), evaluate \(f(x_k)\), the gradient \(g_k\), and a way to apply the Hessian \(H_k\) to vectors.
  2. Use conjugate gradients to find an approximate solution \(p_k\) to \(H_kp=-g_k\). This gives a curvature-aware search direction.
  3. Run a line search along that direction to choose a step length \(\alpha_k\).
  4. Move to \(x_{k+1}=x_k+\alpha_kp_k\), then repeat until the gradient is small or another stopping rule is met.

This separation is important. The inner CG solve proposes a direction based on the local model. The line search checks that the real function actually improves along that direction.

What CG Is Doing Inside

The name "conjugate gradient" can make the method sound like a new optimizer inside the optimizer. For Newton-CG, it is simpler than that. At the current outer point \(x_k\), freeze the gradient and Hessian and look only at the local quadratic model:

\[ m_k(p)=f(x_k)+g_k^\mathsf{T}p+\frac{1}{2}p^\mathsf{T}H_kp. \]

CG is the inner routine that tries to find the bottom of this quadratic model without forming \(H_k^{-1}\). If the Hessian is positive definite, the bottom is where the model gradient is zero:

\[ \begin{aligned} \nabla m_k(p)&=g_k+H_kp=0,\\ H_kp&=-g_k. \end{aligned} \]

That equation says: find a step \(p\) whose Hessian-scaled effect cancels the gradient. A direct Newton method would solve this system with a matrix factorization. CG instead builds \(p\) a little at a time by asking only for products \(H_kv\).

Start With The Residual

Imagine guessing a step \(p_j\). The residual is the amount by which that guess still fails to satisfy the Newton equation:

\[ r_j=-g_k-H_kp_j. \]

If \(r_j=0\), the guess solves \(H_kp=-g_k\) exactly. At the usual starting guess \(p_0=0\), the residual is \(r_0=-g_k\). So the first CG direction is just the downhill direction of the quadratic model. After that, the residual has a second meaning: it points toward the part of the quadratic bowl that the current step has not fixed yet.

Why Not Follow The Residual Every Time?

Plain steepest descent would repeatedly move in the current residual direction. On a skinny elliptical bowl, that often zig-zags: one step fixes too much of one coordinate, the next step partially undoes it, and progress crawls along the valley.

CG keeps a short memory. It starts from the residual, but it adjusts the next search direction so that the new line search does not spoil what earlier line searches already optimized. This is where the word "conjugate" enters.

\[ d_i^\mathsf{T}H_kd_j=0\quad(i\ne j). \]

This is not ordinary perpendicularity. It is perpendicularity in the stretched geometry of the Hessian. On the quadratic model, two \(H_k\)-conjugate directions behave like independent axes: after CG has chosen the best amount to move along \(d_i\), a later move along \(d_j\) does not disturb that earlier best amount.

One Inner CG Step

A single inner CG step has a small, repeatable rhythm. Suppose the current approximate Newton step is \(p_j\), the residual is \(r_j\), and the current search direction is \(d_j\).

  1. Ask for one Hessian-vector product \(H_kd_j\). This is the only place where CG needs Hessian information for this inner step.
  2. Choose a scalar \(\eta_j\) that minimizes the quadratic model along the line \(p_j+\eta d_j\).
  3. Update the approximate Newton step: \(p_{j+1}=p_j+\eta_jd_j\).
  4. Update the residual: \(r_{j+1}=r_j-\eta_jH_kd_j\). If this is small, the Newton equation is solved well enough.
  5. Blend the new residual with the old direction to make the next direction \(d_{j+1}\), chosen to be \(H_k\)-conjugate to earlier directions.

For a positive-definite quadratic, the line-minimizing step length is

\[ \eta_j=\frac{r_j^\mathsf{T}r_j}{d_j^\mathsf{T}H_kd_j}. \]

You do not need to memorize this fraction first. Its job is more important than its form: it picks the best distance to travel along the current direction on the local quadratic model.

Why Newton-CG Stops Early

In exact arithmetic, CG can solve an \(n\)-variable positive-definite quadratic in at most \(n\) inner steps. Newton-CG usually does not wait that long. The outer method only needs a useful descent direction, not a perfect solution of the temporary quadratic model. So the inner loop is commonly stopped when the residual is small enough, when the inner iteration budget is reached, or when the Hessian shows non-positive curvature and the local model is no longer a proper bowl in that direction.

Why Hessian-Vector Products Matter

The phrase "uses the Hessian" can sound expensive, and a dense Hessian really is expensive. For \(N\) variables, a dense Hessian has \(N^2\) entries. At \(N=100{,}000\), storing it directly is already unrealistic for ordinary hardware.

Newton-CG avoids that storage when you provide a Hessian-vector product function. Instead of asking for all entries of \(H_k\), CG repeatedly asks one question:

\[ \text{given a vector } v,\quad \text{return } H_kv. \]

That product can often be computed from structure, sparsity, automatic differentiation, or a matrix-free routine. The optimizer receives the curvature information it needs without ever seeing a dense matrix.

Practical habit: if your Hessian is sparse, structured, or available through automatic differentiation, start by writing hessp. It is often simpler and more scalable than constructing the full Hessian.

Direction Is Not Step Length

The Newton equation gives a direction \(p_k\). It does not guarantee that the full step \(x_k+p_k\) is safe for the original nonlinear objective. The quadratic model is only a local approximation, and it can be overconfident far from \(x_k\).

Newton-CG therefore uses a line search:

\[ x_{k+1}=x_k+\alpha_kp_k, \qquad \alpha_k>0. \]

During that line search, \(x_k\) and \(p_k\) are fixed. The only moving quantity is the scalar \(\alpha\). So the algorithm is really checking a one-dimensional slice of the original objective, \(\phi(\alpha)=f(x_k+\alpha p_k)\), not inventing a new optimization problem.

The tempting shortcut is to accept any \(\alpha\) that makes \(f(x_k+\alpha p_k)

\[ f(x_k+\alpha p_k) \le f(x_k)+c_1\alpha g_k^\mathsf{T}p_k, \qquad 0

The scalar \(c_1\) is the caution setting. Because a descent direction has \(g_k^\mathsf{T}p_k<0\), the right side is below \(f(x_k)\). A small \(c_1\) asks for a modest but nontrivial drop; it filters out steps that technically decrease the objective but barely move. Wolfe-style line searches add a curvature check as well, so the accepted \(\alpha\) is not only safe but also not needlessly tiny.

Meaning: CG chooses a curvature-aware direction in step space. The line search chooses how much of that direction the real nonlinear objective is willing to accept.

Watch It On Rosenbrock

The Rosenbrock function is a classic optimizer stress test:

\[ \begin{aligned} f(x_0,x_1)=100(x_1-x_0^2)^2+(1-x_0)^2, \\ x^\star=(1,1),\quad f(x^\star)=0. \end{aligned} \]

The minimum is easy to state, but the valley is narrow and curved. Gradient-only methods can zig-zag across the valley. Newton-CG uses curvature to aim more directly along the valley while the line search prevents reckless full steps.

In more than two dimensions, Rosenbrock has a banded Hessian. That means each variable interacts only with its neighbors, so a Hessian-vector product can be computed without forming a dense matrix. For an interior coordinate, the product has the pattern

\[ \begin{aligned} [H(x)p]_i &= -400x_{i-1}p_{i-1}\\ &\quad +(202+1200x_i^2-400x_{i+1})p_i\\ &\quad -400x_ip_{i+1}. \end{aligned} \]

Read that formula like code: the \(i\)-th output uses the previous, current, and next entries of \(p\). That local dependency is exactly the kind of structure Newton-CG can exploit.

Use Newton-CG In SciPy

In SciPy, call minimize with method="Newton-CG". Provide an analytic gradient through jac. When possible, provide hessp so the optimizer can apply curvature without materializing the full Hessian. 

import numpy as np
from scipy.optimize import check_grad, minimize

def rosen(x):
    return np.sum(100.0 * (x[1:] - x[:-1]**2.0)**2.0 + (1 - x[:-1])**2.0)

def rosen_der(x):
    der = np.zeros_like(x)
    der[1:-1] = (
        200 * (x[1:-1] - x[:-2]**2)
        - 400 * x[1:-1] * (x[2:] - x[1:-1]**2)
        - 2 * (1 - x[1:-1])
    )
    der[0] = -400 * x[0] * (x[1] - x[0]**2) - 2 * (1 - x[0])
    der[-1] = 200 * (x[-1] - x[-2]**2)
    return der

def rosen_hess_p(x, p):
    hp = np.zeros_like(x)
    hp[0] = (1200 * x[0]**2 - 400 * x[1] + 2) * p[0] - 400 * x[0] * p[1]
    hp[1:-1] = (
        -400 * x[:-2] * p[:-2]
        + (202 + 1200 * x[1:-1]**2 - 400 * x[2:]) * p[1:-1]
        - 400 * x[1:-1] * p[2:]
    )
    hp[-1] = -400 * x[-2] * p[-2] + 200 * p[-1]
    return hp

x0 = np.array([1.3, 0.7, 0.8, 1.9, 1.2])

# A small value here is a strong sign that rosen_der matches rosen.
print("gradient check:", check_grad(rosen, rosen_der, x0))

result = minimize(
    rosen,
    x0,
    method="Newton-CG",
    jac=rosen_der,
    hessp=rosen_hess_p,
    options={"xtol": 1e-8, "disp": True},
)

print(result.x)
print(result.fun)
print(result.success, result.message)

The key function is rosen_hess_p(x, p). It does not return the Hessian. It returns the product \(H(x)p\). That is the contract Newton-CG needs for its inner CG loop.

When To Use It

Use Newton-CG when Prefer another method when
You have a smooth unconstrained objective and a reliable gradient. You only have function values; start with derivative-free methods such as Nelder-Mead or Powell.
You can compute Hessian-vector products more cheaply than storing a dense Hessian. The problem is small and a dense Hessian factorization is cheap; direct Newton or trust-exact may need fewer outer iterations.
The variable count is large enough that \(O(N^2)\) Hessian storage is unattractive. The objective is noisy, nonsmooth, or has unreliable gradients; the curvature model will be hard to trust.
You want second-order directions with line-search safeguards. The Hessian is often indefinite or the line search struggles; trust-ncg or trust-krylov can be more robust.

The most common Newton-CG disappointments are practical rather than mysterious: wrong gradients, inconsistent Hessian-vector products, badly scaled variables, or expecting the local quadratic model to be reliable far from the current point.

Debugging habit: if Newton-CG behaves strangely, first check that jac matches the objective, then check that hessp(x, p) matches finite differences of the gradient in the direction \(p\). A sign error in either one can make a good algorithm look chaotic.
