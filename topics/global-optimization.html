Bounded many-basin optimization

Global optimization tutorial

Global optimizers search a bounded domain when one local run is too dependent on the starting point. This tutorial starts from the local minimizer failure mode, then compares SciPy's global search families on the eggholder surface.

From One Basin To Many

A local minimizer starts from one point and tries to improve nearby. That is exactly what you want when the surface has one obvious basin or when your starting point is already good. But on a rough surface, "nearby improvement" can trap the optimizer in a basin that is low compared with its neighbors but not low compared with the whole domain.

Global optimization adds the bounded search domain to the problem. The variable \(x\) must stay inside a box \(\Omega\). Each coordinate has a lower bound \(\ell_i\) and an upper bound \(u_i\):

\[ \begin{aligned} \min_{x\in\Omega}\quad & f(x),\\ \Omega &= \{x:\ell_i\leq x_i\leq u_i\}. \end{aligned} \]

The global promise is stronger than local stationarity. A true global minimizer \(x^\star\) should beat every feasible point in the bounded domain:

\[ f(x^\star)\leq f(x) \quad\text{for all }x\in\Omega. \]

The tempting shortcut is to run a local optimizer once and trust the answer. That can be fine on a simple surface. On a many-basin surface, it mainly tells you which basin the starting point fell into.

The Eggholder Landscape

The SciPy tutorial uses the eggholder function because it makes the global-search problem visible. The input has two coordinates, \(x_0\) and \(x_1\), both bounded between \(-512\) and \(512\). The surface has many valleys, ridges, and local minima. Writing the repeated pieces as \(a\), \(b\), and \(c\) makes the shape easier to read:

\[ \begin{aligned} a &= x_1 + 47,\\ b &= x_0/2 + x_1 + 47,\\ c &= x_0 - a,\\ f(x_0,x_1) &= -a\sin\sqrt{|b|} - x_0\sin\sqrt{|c|}. \end{aligned} \]

The formula is less important than the shape. A small move can leave one valley and enter another. A local optimizer may do everything right inside one basin and still miss a much lower basin elsewhere.

Why Local Search Is Not Enough

A local optimizer can stop at a point \(\hat{x}\) where the local slope is nearly zero. That is a local success signal, not a global one:

\[ \begin{aligned} \nabla f(\hat{x})&\approx 0,\\ f(\hat{x})&\neq \min_{x\in\Omega} f(x) \quad\text{in general}. \end{aligned} \]

Another tempting shortcut is to launch many random local starts and take the best one. That is often a useful baseline, but it is still a sampling strategy. If the best basin is small or hard to enter, a pile of local runs may miss it.

Global Search Families

There is no universally best global optimizer because "explore the whole box" can mean several different things. SciPy's global methods differ by how they choose candidate points before optional local polishing.

Cover The Domain Deterministically

shgo and direct are deterministic bounded search methods. shgo samples the domain in a structured way and can report multiple local minima. direct repeatedly subdivides rectangles, balancing large unexplored regions against regions with promising function values.

Use A Population

differential_evolution keeps a population of candidate points. New candidates are made by combining existing points. The failed shortcut it avoids is trusting one trajectory too much: a population can compare several basins at once.

Jump Between Basins

dual_annealing uses stochastic moves controlled by a temperature-like schedule, then can polish with local search. basinhopping makes random perturbations and then runs a local minimizer from the perturbed point. These methods are useful when escaping a basin matters more than following one smooth path.

Use Global Optimization In SciPy

The global optimizers live in scipy.optimize. Most need finite bounds because their job is to explore a box. Several can polish the best point with a local minimizer after the broad search. 

import numpy as np
from scipy.optimize import (
    basinhopping,
    differential_evolution,
    direct,
    dual_annealing,
    shgo,
)

def eggholder(x):
    a = x[1] + 47
    b = x[0] / 2 + x[1] + 47
    c = x[0] - a
    r1 = np.sqrt(abs(b))
    r2 = np.sqrt(abs(c))
    return (
        -a * np.sin(r1)
        - x[0] * np.sin(r2)
    )

bounds = [
    (-512, 512),
    (-512, 512),
]
diff_evo = differential_evolution

shgo_result = shgo(
    eggholder,
    bounds,
    n=96,
    iters=2,
    sampling_method="sobol",
)

dual_result = dual_annealing(
    eggholder,
    bounds,
    seed=2,
)

diff_result = diff_evo(
    eggholder,
    bounds,
    seed=1,
)

direct_result = direct(
    eggholder,
    bounds,
)

hop_result = basinhopping(
    eggholder,
    x0=np.array([0.0, 0.0]),
    minimizer_kwargs={
        "method": "L-BFGS-B",
        "bounds": bounds,
    },
    seed=4,
)

results = [
    ("shgo", shgo_result),
    ("dual", dual_result),
    ("diff_evo", diff_result),
    ("direct", direct_result),
    ("basinhopping", hop_result),
]

for name, result in results:
    value = result.fun
    point = result.x
    print(name, value, point)

A stochastic method's seed controls one run, not mathematical certainty. For important black-box problems, compare several seeds, inspect the best basins, and then polish or validate the finalists with domain-specific checks.

When To Use It

Use global optimization when Prefer local optimization when
The objective is bounded and has many plausible local minima. The surface is smooth, convex-like, or already near a good basin.
Function evaluations are expensive, but a wrong basin is even more expensive. You need fast refinement after a good initial point is known.
You can afford multiple searches, stochastic seeds, or deterministic space coverage. The dimension is high enough that broad space exploration becomes impractical.
You need evidence about competing basins, not just one local optimum. You need proof for a convex or structured problem; use the solver family that matches that structure.
Practical habit: use global optimization to identify promising basins, then inspect the best solutions with local minimizers and domain constraints. The result object is evidence, not a proof of globality.
