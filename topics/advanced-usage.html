Advanced optimizer interfaces

Advanced optimizer interfaces tutorial

This page is not another algorithm family. It explains how SciPy lets you connect your own minimizer to the usual minimize interface, and how workers parallelizes batches of independent objective evaluations.

Start With The Shared Interface

Most pages in this guide choose a solver by passing a string: method="BFGS", method="SLSQP", or method="trust-krylov". That feels like the method is the whole story, but minimize is really a contract around three objects.

Object Meaning Typical shape
fun The objective function to minimize. fun(x, *args) -> float
x0 The starting point supplied by the user. one-dimensional array
res The result object returned by the solver. OptimizeResult

The advanced features on this page keep that contract and swap out the machinery behind it. A custom minimizer changes who decides the next point. workers changes where independent objective calls are evaluated. Neither feature changes the meaning of fun, x0, or the returned result.

Why A Custom Minimizer Exists

Suppose you already have a domain solver: maybe a simulator wrapper, a research prototype, or a tiny coordinate search you want to use inside basinhopping. The tempting shortcut is to call that solver outside SciPy and translate its answer by hand.

That shortcut works once, but it loses the common SciPy interface. Meta-optimizers cannot call your method. Existing callback code may not run. Result handling becomes a special case. Passing a callable as method solves that by making your solver look like any other minimize method.

method(fun, x0, args, **kwargs, **options) -> OptimizeResult

Name each part before using it. fun is the objective with the usual SciPy calling convention. x0 is the initial point. args are fixed extra objective parameters. kwargs are other pieces passed through minimize, such as jac, callback, bounds, or constraints. options are the entries from the options dictionary, expanded into keyword arguments.

Practical contract: accept **options even when you ignore most of it. Future SciPy versions and SciPy wrappers may pass additional keywords, and a custom method should not fail just because it received an option it does not need.

Return A Result, Not Just A Point

A minimizer finds a point, but callers need more than the point. They need to know whether the search stopped normally, how much work was done, and what objective value was reached. That is why the custom method should return an OptimizeResult rather than a raw array.

Field Question it answers
x Which point did the solver report?
fun What is the objective value at that point?
success Did the solver believe it exited successfully?
message Why did it stop?
nfev, nit How many function evaluations and iterations were used?

Different SciPy solvers expose different fields, so do not assume every result contains every possible attribute. The important habit is to return the fields your custom method really knows. A tiny coordinate search can count objective calls and iterations. It should not pretend to know a Hessian inverse.

One Custom Iteration, Slowly

The coordinate-search example has a deliberately humble model of progress. At the current best point, it tries moving one coordinate down and one coordinate up by a fixed stepsize. If the objective improves, that trial becomes the new best point.

  1. Copy x0 into a floating point working vector.
  2. Evaluate fun(x, *args) once.
  3. For each coordinate-sign pair, test one trial step.
  4. Keep the trial only if it lowers the objective.
  5. Call callback(x) after an accepted improvement.
  6. Stop when one full pass finds no improvement.

This is not a recommendation to use coordinate search for serious smooth problems. It is a small enough algorithm that the interface is visible. The method receives the same objective that a built-in SciPy method would receive, and it returns the same kind of result object.

What Workers Parallelize

Parallel execution helps only when the optimizer has a batch of independent objective evaluations. It does not make one dependent step independent. The outer loop of an optimizer still waits until the needed evaluations are done before choosing the next iteration.

In differential_evolution, one generation contains many candidate solutions whose objective values can be evaluated independently. In selected local minimizers, finite-difference gradients create a smaller independent batch around one point.

D_i f(x) ~= (f(x + h e_i) - f(x)) / h, for i = 1, ..., N

For a two-point forward-difference gradient, the optimizer needs the baseline value \(f(x)\) plus one perturbed value for each coordinate: \(f(x+h e_1)\), \(f(x+h e_2)\), and so on. That is the \(N+1\) evaluation batch. If each call is expensive and independent, several workers can evaluate the batch at the same time.

Workers Are A Map-Like Contract

A worker does not receive a magic optimizer object. It receives a function and an iterable of candidate vectors, then returns objective values in the same order. Conceptually, the contract is

workers(func, iterable) -> values

If workers is an integer, SciPy can create a multiprocessing pool for you. If workers is a callable, SciPy uses it as the map-like object. A reusable pool.map is often better when process startup overhead matters.

The failed shortcut is assuming workers=4 is always faster. It can be slower when the objective is cheap, when large data must be serialized, or when the objective cannot be pickled. For differential_evolution, parallel workers also require deferred population updating, because the best member cannot be updated continuously while candidate values are still running.

Mental model: workers improve throughput for a batch of independent function calls. They do not improve the mathematical search model, fix scaling, or reduce the number of iterations by themselves.

Use The Interfaces In SciPy

The code below keeps the lines short on purpose. The custom method accepts extra options, returns an OptimizeResult, and can be passed directly as method. The worker examples reuse one pool for both a population solver and a local method that may use finite differences. 

import numpy as np
from multiprocessing import Pool
from scipy.optimize import (
    OptimizeResult,
    differential_evolution,
    minimize,
    rosen,
)

def custmin(
    fun,
    x0,
    args=(),
    stepsize=0.1,
    maxiter=100,
    callback=None,
    **options,
):
    x = np.array(x0, dtype=float)
    fx = fun(x, *args)
    cb = callback
    step = stepsize
    nfev = 1
    nit = 0
    moves = []

    for i in range(x.size):
        moves += [(i, -1), (i, 1)]

    for k in range(maxiter):
        improved = False
        for i, s in moves:
            xp = x.copy()
            xp[i] += s * step
            fp = fun(xp, *args)
            nfev += 1
            if fp >= fx:
                continue
            x = xp
            fx = fp
            improved = True
            if cb:
                cb(x)
        if not improved:
            break
        nit = k + 1

    return OptimizeResult(
        x=x,
        fun=fx,
        success=True,
        message="stopped",
        nfev=nfev,
        nit=nit,
    )

x0 = np.array([1.3, 0.7, 0.8])
custom = minimize(
    rosen,
    x0,
    method=custmin,
    options={"stepsize": 0.05},
)

bounds = [(0.0, 10.0)] * 3
with Pool(2) as pool:
    de = differential_evolution(
        rosen,
        bounds,
        workers=pool.map,
        updating="deferred",
    )
    worker_opts = {
        "workers": pool.map,
    }
    lbfgsb = minimize(
        rosen,
        x0,
        method="L-BFGS-B",
        options=worker_opts,
    )

Notice the two kinds of customization. method=custmin replaces the algorithm that chooses trial points. The workers option keeps the algorithm but changes how a batch of objective evaluations is executed.

When To Use It

Feature Use when Watch for
Callable method You need to wrap a solver, prototype a method, or plug a method into a SciPy meta-optimizer. Preserve the SciPy calling convention and return a useful OptimizeResult.
OptimizeResult Callers need a standard way to inspect the point, objective value, status, and work counts. Only report fields the method actually knows.
workers=int You want SciPy to manage process-based parallel evaluation. The objective must be pickleable, and startup overhead can dominate cheap functions.
Map-like workers You want to reuse a pool, route work to another executor, or batch evaluations in a custom way. The callable must behave like map: same input order, same output order.
Practical habit: customize the interface only when it removes a real integration problem. If a built-in method and a serial objective are already clear and fast, adding a custom method or worker pool mostly adds moving parts.
